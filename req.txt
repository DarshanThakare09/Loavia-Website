1. Project Overview
LOAVIA is a premium cookie brand with the tagline 'Healthy Inside, Yummy Outside.' This document defines the complete requirements for building LOAVIA's web platform — a brand website combined with a fully functional e-commerce system. The platform must serve as both a storytelling vehicle for the brand and a high-conversion shopping experience.
Mission: Build a high-conversion, premium cookie e-commerce platform that combines fast ordering, emotional brand engagement, product customization, and a scalable backend architecture.
The website must feel: Fast (low friction) | Visual (product-first) | Emotional (cravings-driven) | Interactive (micro-engagements)
1.1 Business Objectives
Drive impulse purchases through visual, emotion-first design
Enable gifting use-cases with occasion-based customization
Provide an interactive brand experience through animations and 3D elements
Build repeat customers through a loyalty and reward points system
Maintain a scalable, API-first backend architecture ready for future mobile app integration
1.2 Design Philosophy
Premium, luxury, and modern UI — no generic or template-looking layouts
Brand-focused color palette: warm browns, creams, golds — derived from the LOAVIA logo
Mobile-first and fully responsive across all device sizes
High-quality product imagery with smooth, purposeful animations (GSAP scroll effects, Three.js 3D cookies)
Fast loading speed — optimized images, lazy loading, performance-first code

4. Core System Logic & Backend Requirements
The following describes the exact business logic that the backend must implement. This is non-negotiable system behavior.
4.1 Authentication System
User registration: name, email (unique index), phone, password (Bcrypt/Argon2 hashed), role (user/admin)
Login returns a signed JWT. Token stored client-side (httpOnly cookie recommended)
Password reset via email OTP or reset link
All authenticated routes validate JWT on every request
4.2 Product & Pricing Logic
Each product stores: product_id (UUID), name, slug, description, ingredients (Array), category_id, image_urls (Array), base_price, discount_pc (0–100), stock_qty, is_active, average_rating
Pricing formula (calculated at application layer before serving API):
     final_price = base_price − (base_price × discount_pc / 100)
Availability: status = IN_STOCK if (stock_qty > 0 AND is_active = true), else OUT_OF_STOCK
Time-based discounts: start_date and end_date fields. Backend middleware checks if current_time is within the range before applying discount_pc
If discount_pc is invalid (< 0 or > 100), return HTTP 400 Bad Request
4.3 Cart System
Cart persisted in Redis (for speed) or DB (for long-term), linked to user_id or session_id
line_item_total = final_price × quantity
cart_subtotal = sum of all line_item_totals
Constraint: if requested_qty > current_stock_qty → return error 'Insufficient Stock'
Guest Cart Merge: when a guest user logs in, their session cart items must be merged with their stored DB cart (no items lost)
4.4 Checkout & Order Flow
The checkout must follow this exact transactional sequence:
Validate all product_ids — confirm they exist and are_active
Inventory Soft-Lock — UPDATE stock_qty = stock_qty - qty WHERE id = X AND stock_qty > 0
Capture and validate delivery address (regex on ZIP / phone)
Apply offers / coupon codes
Send cart_total to Razorpay / Stripe payment gateway
On payment.success webhook → set order status to PAID, hard-deduct inventory
On payment.failure or timeout → Release soft-lock: restore stock_qty
CRITICAL: Price at checkout must be locked at the moment the checkout session starts (with a TTL). Admin price changes during active checkout sessions must not affect the in-progress order.
4.5 Order Management System (OMS)
Order entity fields: order_id, user_id, items (JSON), total_amount, tax, shipping_fee, payment_ref, order_status (Enum)
Order status flow:
PENDING → PAID → PREPARING → SHIPPED → DELIVERED → CANCELLED
Each status transition records a timestamp (placed_at, shipped_at, delivered_at, etc.)
Admin can manually override order status
Cancellation triggers refund logic and stock restoration
4.6 Review & Rating System
Verified Buyer Gate: A user may only leave a review if:
     SELECT COUNT(*) FROM Orders WHERE user_id = X AND product_id = Y AND status = 'DELIVERED' > 0
Review data: rating (1–5), comment text, timestamp, is_verified flag
Average rating cached in the Products table (updated on each new review) — do NOT recalculate on every page load
4.7 Loyalty & Rewards System
Users earn points on every completed order (e.g., 1 point per ₹10 spent)
Points can be redeemed at checkout as a discount
Points ledger: user_id, points_earned, points_redeemed, balance, transaction_reference
4.8 Recommendation Engine
Content-Based: Calculate Jaccard Similarity between product ingredient arrays. If user views 'Dark Chocolate Sea Salt', recommend products sharing 'Dark Chocolate' in ingredients
Collaborative Filtering: Query Orders table for most frequent product pairs associated with the current item's category ('Users who bought X also bought Y')
4.9 Race Condition & Edge Case Handling
These are critical and must be implemented correctly:
Two users buying the last item simultaneously: use Database Transactions with SELECT FOR UPDATE or Optimistic Concurrency Control
Out-of-stock product: disable 'Add to Cart' button via API response field can_purchase: false
Price change during active checkout: honor price at checkout session start (TTL-based price lock)
Payment timeout: automatically release soft-locked inventory

7.2 Hero Section
Full viewport height
3D cookie model (Three.js) — slow rotation or floating particle effect
Bold headline + supporting tagline
'Shop Now' CTA — hover effect (elevate / color shift)
GSAP entrance animation on page load (stagger text, fade-in cookie)
7.3 Product Cards
Consistent card sizing across all grids
Hover: image subtle zoom/shift, 'Add to Cart' or 'Quick View' button appears
Smooth transition — no jarring layout shift
7.4 Scroll Animations (GSAP ScrollTrigger)
Sections fade/slide in as user scrolls down
Product cards stagger-animate into view
Testimonials section horizontal scroll or fade carousel
Animations must not block or slow page performance
7.5 Loading & Transitions
Light fade-in on initial page load — no abrupt flash
Page route transitions (smooth, 200–300ms)
Skeleton loaders for product grids while data fetches
7.6 Color Palette
Derive from the LOAVIA logo:
Primary: Deep Brown (#5C3317)
Accent / Gold: #A0772A
Background Cream: #F5ECD7
Text: Near-black (#1A1A1A), Secondary gray (#555555)
Error: Standard red. Success: Standard green
7.7 Typography
Brand headings: Elegant serif (e.g., Playfair Display or similar premium serif from Google Fonts)
Body text: Clean, readable sans-serif (e.g., Inter or DM Sans)
All text must pass WCAG 2.1 AA contrast requirements
8. Payment Integration
Primary gateway: Razorpay (UPI, Credit/Debit Cards, Net Banking, Wallets)
Optional secondary: Stripe (for international orders)
SSL certificate required — all payment data over HTTPS only
Automatic invoice generation (PDF) on order confirmation
Payment success notification: in-app confirmation page + email
Payment failure notification: user-facing error message + retry option
Webhook handler: /api/payment/webhook must process payment.success and payment.failed events from gateway
On success: set order PAID, hard-deduct inventory, trigger notification
On failure: release soft-lock, restore stock, notify user
9. Notifications
Email notifications: order confirmation, order status updates, password reset
SMS notifications: order confirmation, shipping update (optional / phase 2)
In-app: cart badge count, order status timeline on tracking page
Admin alerts: low stock warning email when stock_qty < 10 (automated)
